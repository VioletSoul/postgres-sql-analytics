## Назначение `wal_analytics.sql`

`wal_analytics.sql` — диагностический запрос для PostgreSQL 15.x, который даёт целостное представление о состоянии подсистемы WAL кластера.  
Он помогает быстро понять:

- сколько WAL было сгенерировано с момента последнего сброса статистики
- какова текущая скорость генерации WAL
- как работает архивирование WAL
- есть ли проблемы с репликацией (lag на репликах)
- удерживают ли replication‑слоты слишком много WAL и создаётся ли риск заполнения диска

Запрос не привязан к конкретной схеме или базе данных и анализирует состояние на уровне всего сервера.

---

## Источники данных

Отчёт использует стандартные системные представления PostgreSQL:

- `pg_stat_wal` – статистика генерации WAL
- `pg_stat_archiver` – состояние архиватора WAL
- `pg_stat_replication` – состояние потоковой репликации и lag по каждой реплике
- `pg_replication_slots` – информация о replication‑слотах и удерживаемом ими WAL

---

## Общая структура

Запрос построен как серия CTE (WITH‑блоков), каждый из которых отвечает за свою часть картины.

### 1. `wal_stat`

Снимает «моментальный снимок» текущей статистики WAL:

- текущее время и момент `stats_reset`
- общее количество записей в WAL (`wal_records`)
- количество full page images (`wal_fpi`)
- суммарный объём сгенерированного WAL в байтах (`wal_bytes`)
- счётчики, связанные с заполнением WAL‑буферов и операциями записи/синхронизации

Это сырые накопительные счётчики, которые PostgreSQL ведёт с момента последнего сброса статистики.

### 2. `wal_rate`

На основе `wal_stat` вычисляет:

- время, прошедшее с момента `stats_reset`
- среднюю скорость генерации WAL в байтах в секунду (`wal_bytes_per_second`)

Эти показатели позволяют оценить текущую интенсивность записи и планировать необходимый объём диска под `pg_wal`.

### 3. `wal_archiver`

Собирает состояние архиватора WAL:

- количество успешно архивированных сегментов (`archived_count`)
- количество неудачных попыток архивирования (`failed_count`)
- последний успешно архивированный сегмент и время (`last_archived_wal`, `last_archived_time`)
- последний проблемный сегмент и время (`last_failed_wal`, `last_failed_time`)

На основе этих полей можно оценить стабильность архивации и наличие незакрытых ошибок.

### 4. `wal_replication` и `wal_replication_summary`

`wal_replication` считывает по каждой реплике:

- `application_name`, адрес клиента
- состояние соединения (`state`) и тип синхронизации (`sync_state` — sync/async)
- lag по LSN в байтах: `sent_lag_bytes`, `write_lag_bytes`, `flush_lag_bytes`, `replay_lag_bytes`
- при наличии — временной lag: `write_lag`, `flush_lag`, `replay_lag`

`wal_replication_summary` агрегирует эти данные:

- общее количество реплик
- сколько из них не в состоянии `streaming`
- количество синхронных и асинхронных реплик
- максимальные значения lag по `replay`, `write` и `flush` среди всех стендбаев

Так можно быстро увидеть, есть ли реплики с существенным отставанием.

### 5. `wal_slots` и `wal_slots_summary`

`wal_slots` анализирует replication‑слоты:

- `slot_name`, `slot_type` (`physical` / `logical`)
- активен ли слот (`active`)
- статус WAL для слота (`wal_status`: `normal`, `reserved`, `extended`, `lost` и т.п.)
- объём WAL, который удерживается слотом и пока не может быть удалён (`retained_wal_bytes`)

`wal_slots_summary` даёт сводку по слотам:

- общее число слотов
- количество неактивных слотов
- число слотов в состояниях `lost`, `reserved`, `extended`
- максимальный объём удерживаемого WAL среди всех слотов

Это позволяет вовремя заметить ситуацию, когда забытый слот копит гигабайты WAL и создаёт риск переполнения диска.

### 6. `summary` — сводка и флаги риска

CTE `summary` объединяет все предыдущие блоки и формирует:

- ключевые числовые показатели (объём WAL, скорость генерации, количество реплик/слотов, максимальные лаги и т.п.)
- агрегированные метрики lag
- несколько булевых флагов риска:

    - `archiver_has_recent_errors` – есть свежие ошибки архивации, которые не были «перекрыты» успешной попыткой
    - `replication_lag_risk` – максимальный lag по репликации превышает заданный порог (по байтам)
    - `slot_retention_risk` – объём WAL, удерживаемый хотя бы одним слотом, превышает заданный порог

## Итоговый результат

Финальный `SELECT` возвращает единую таблицу, в которой строки разделяются по значению в колонке `section`:

- `section = 'summary'` – одна агрегированная строка с общей картиной по кластеру
- `section = 'replica'` – по одной строке на каждую реплику (lag, состояние, тип синхронизации)
- `section = 'slot'` – по одной строке на каждый replication‑слот (объём удерживаемого WAL и статус)

Колонка `name` содержит:

- для `summary` – `NULL`
- для `replica` – `application_name` реплики
- для `slot` – `slot_name` слота

Часть колонок осмысленно заполнена только для `summary` (архиватор, сводные счётчики); в строках‑репликах и строках‑слотах в этих полях стоят `NULL`.  
Это позволяет иметь **одну схему результата**, но логически три блока информации.

---

## Типичные сценарии использования

- Быстрая проверка состояния WAL и репликации во время инцидента
- Регулярный ручной аудит состояния кластера (после обновлений, изменений конфигурации и т.п.)
- Источник данных для дашборда: поле `section` удобно использовать для разбивки на панели
- Оценка риска заполнения диска из‑за lag’а реплик или чрезмерного удержания WAL replication‑слотами

---

## Как интерпретировать ключевые поля

- `wal_generated_total` (`summary`) – общий объём WAL, сгенерированный с момента `stats_reset`
- `wal_bytes_per_second` / `wal_rate_human` – средняя скорость генерации WAL; полезно для capacity planning
- `max_replay_lag`, `max_write_lag`, `max_flush_lag` – максимальные лаги среди всех реплик; рост этих значений указывает на проблемы с репликацией или I/O
- `max_retained_wal` (`summary`) и `wal_generated_total` для `section = 'slot'` – объём WAL, удерживаемый слотами; помогает оценить корректность работы со слотами
- `archiver_has_recent_errors`, `replication_lag_risk`, `slot_retention_risk` – высокоуровневые флаги, показывающие, куда смотреть в первую очередь при разборе проблем
