## Назначение `query_analytics.sql`

`query_analytics.sql` — диагностический запрос для PostgreSQL 15.x, который даёт сводную картину по выполненным SQL‑операторам на основе `pg_stat_statements`.  
Он помогает ответить на вопросы:

- какие запросы тратят больше всего времени суммарно
- какие запросы медленные по средней длительности
- какие запросы сильнее всего нагружают диск (shared/local/temp I/O)
- где низкая эффективность кэша
- есть ли запросы с чрезмерными затратами на планирование

Запрос работает на уровне кластера и не привязан к конкретной схеме.

---

## Источник данных

Основой служит представление `pg_stat_statements`, которое агрегирует статистику планирования и выполнения SQL‑операторов:

- количество вызовов (`calls`)
- суммарное и среднее время выполнения (`total_exec_time`, `mean_exec_time`)
- суммарное и среднее время планирования (`total_plan_time`, `mean_plan_time`)
- количество возвращённых строк (`rows`)
- работа с буферами: `shared_blks_*`, `local_blks_*`, `temp_blks_*`
- время чтения/записи блоков (`blk_read_time`, `blk_write_time`)

Дополнительно используется информация из `pg_roles` и `pg_database` для привязки к пользователю и базе.

---

## Общая структура

Запрос построен как цепочка CTE (WITH‑блоков):

1. `query_base`
2. `query_enriched`
3. `query_ranked`
4. `query_diagnostics`
5. `query_summary`
6. `top_queries`
7. финальный `SELECT` с `UNION ALL`

Каждый блок добавляет свой слой агрегации и метаданных.

---

## 1. `query_base`

Базовый слой, напрямую читающий `pg_stat_statements`:

- подтягивает:
    - идентификаторы: `userid`, `dbid`, `queryid`
    - исходный текст запроса (`query`)
    - планы и времена планирования/выполнения
    - счётчики блоков и времён I/O
- джойнится с:
    - `pg_roles` → `user_name`
    - `pg_database` → `database_name`
- отфильтровывает:
    - служебные БД (`template0`, `template1`)
    - собственные запросы к `pg_stat_statements`, чтобы не засорять статистику

Результат — «сырая» статистика по запросам, чуть обогащённая именами пользователя и базы.

---

## 2. `query_enriched`

На этом шаге добавляются производные метрики и классификация:

- `normalized_query_preview`
    - текст запроса с схлопнутыми пробелами и обрезкой до 500 символов
    - удобно для просмотра в таблице/дашборде

- метрики нагрузки:
    - `total_shared_block_access` — суммарные обращения к shared‑буферам
    - `total_local_block_access` — к локальным буферам
    - `total_temp_block_activity` — активность с временными блоками

- эффективность кэша:
    - `shared_cache_hit_ratio_percent` = share hit / (hit + read)

- средние показатели:
    - `avg_exec_time_ms` — среднее время выполнения за вызов
    - `avg_plan_time_ms` — среднее время планирования
    - `avg_rows_per_call` — среднее количество строк на вызов
    - `avg_shared_reads_per_call` — средние логические чтения
    - `avg_temp_written_per_call` — средняя запись во временные блоки

- относительные доли времени:
    - `planning_to_execution_percent` — какая доля времени уходит на планирование
    - `read_io_time_percent` / `write_io_time_percent` — доля времени чтения/записи блоков

- тип запроса:
    - `query_type` — простая классификация по началу текста (`SELECT`, `INSERT`, `UPDATE`, `DELETE`, `VACUUM`, `ANALYZE`, `CREATE`, `ALTER`, `DROP`, `OTHER`)

Это превращает сырую статистику в более человекочитаемый профиль каждого queryid.

---

## 3. `query_ranked`

Добавляются ранги по ключевым осям:

- `rank_total_exec_time` — по суммарному времени выполнения
- `rank_mean_exec_time` — по средней длительности
- `rank_calls` — по числу вызовов
- `rank_shared_reads` — по числу прочитанных shared‑блоков
- `rank_temp_written` — по объёму временной записи

Ранги удобны, если нужно быстро найти «топ проблемных».

---

## 4. `query_diagnostics`

Здесь происходит диагностика и формирование рекомендаций.

### `diagnostic_flag`

Категоризирует запросы по набору правил:

- `FREQUENT_AND_SLOW`
    - много вызовов (`calls >= 1000`) и при этом высокая средняя задержка (`mean_exec_time >= 50 ms`)

- `SLOW_QUERY`
    - высокий `mean_exec_time` (≥ 500 ms), даже если запрос не вызывается слишком часто

- `HEAVY_SHARED_READ`
    - большой объём чтения shared‑блоков (`shared_blks_read` ≥ 100000)

- `TEMP_SPILL_RISK`
    - значительный объём записи во временные блоки (`temp_blks_written` ≥ 10000)

- `LOW_CACHE_EFFICIENCY`
    - низкий hit ratio по shared‑кэшу (< 90%), при этом немалый объём чтений

- `PLANNING_OVERHEAD`
    - доля времени на планирование > 30% от `total_exec_time` и запрос вызывается чаще 100 раз

- `NORMAL`
    - если ни одно из условий выше не сработало

### `recommendation`

Для каждого флага формируется короткое текстовое пояснение:

- `FREQUENT_AND_SLOW` → «частый и медленный, посмотреть план и индексы»
- `SLOW_QUERY` → «высокая средняя задержка, проверить план/джойны/фильтрацию»
- `HEAVY_SHARED_READ` → «много чтений, проверить покрытие индексами и паттерн доступа»
- `TEMP_SPILL_RISK` → «много временных файлов, пересмотреть work_mem и планы сортировки/хэшей»
- `LOW_CACHE_EFFICIENCY` → «низкая эффективность кэша, проверить локальность доступа и размер таблиц»
- `PLANNING_OVERHEAD` → «слишком много времени тратится на планирование, посм. на повторное использование планов и нормализацию запросов»
- иначе → «No immediate performance risk detected.»

---

## 5. `query_summary`

Агрегированная сводка по всему набору запросов:

- `query_count` — сколько разных запросов (queryid)
- `flagged_query_count` — сколько из них имеют флаг `<> 'NORMAL'`
- `total_calls` — суммарное число вызовов
- `total_exec_time_ms` — суммарное время выполнения всех запросов (мс)
- `avg_mean_exec_time_ms` — средняя по всем `mean_exec_time`
- `total_shared_blks_read` / `total_shared_blks_hit` — суммарные чтения/попадания в кэш
- `total_temp_blks_written` — суммарные временные записи
- `max_mean_exec_time` / `max_total_exec_time` — самые «тяжёлые» запросы по средней и суммарной длительности

Эта строка удобна для быстрой оценки «масштаба боли» в системе.

---

## 6. `top_queries`

Выбирает top‑N (по умолчанию 100) запросов из `query_diagnostics`:

- сортировка по `total_exec_time` (основной критерий)
- при равенстве — по `calls`

Это основная рабочая выборка для детального анализа.

---

## 7. Финальный результат

Финальный `SELECT` возвращает две группы строк в одной таблице:

- `section = 'summary'`
    - одна строка со сводной статистикой (`query_summary`)

- `section = 'query'`
    - до 100 строк по отдельным запросам (`top_queries`), с:
        - базовой информацией (база, пользователь, `queryid`, тип запроса)
        - метриками вызовов и времени
        - I/O‑показателями
        - флагами диагностики и текстовыми рекомендациями
        - укороченным текстом запроса (`normalized_query_preview`)

Часть колонок осмысленно заполнена только для `summary`, часть — только для `query`. Для «несвоих» полей используются `NULL`, чтобы схема результата была единой.

---

## Типичные сценарии использования

- Быстрая диагностика «кто грузит базу» по total execution time.
- Поиск одиночных очень медленных запросов (высокий `mean_exec_time`).
- Обнаружение запросов, сильно нагружающих диск (по shared/temp I/O).
- Поиск проблем с планированием (большая доля `total_plan_time`).
- Подготовка отчётов для команды разработки: список запросов с флагами и рекомендациями.

---

## Как интерпретировать ключевые поля

- `diagnostic_flag` / `recommendation` — первичный сигнал и подсказка, куда смотреть.
- `calls` + `mean_exec_time` — «частый и медленный» против «редкий, но катастрофический».
- `shared_blks_read` / `shared_blks_hit` и `shared_cache_hit_ratio_percent` — эффективность работы с кэшем.
- `temp_blks_written` — признак тяжёлых сортировок/хэш‑операций, возможно требует настройки work_mem.
- `planning_to_execution_percent` — если велик, запросы могут страдать от лишнего перепланирования (особенно если часто вызываются с разными литералами вместо параметров).
